# M3：检索基线与检索服务

## 本模块回答什么问题

M3 回答的是：

> 这个项目到底有没有一个可信的检索环境，而不是只在 README 里写了“会搜索”？

如果没有稳定的检索基线，后面的 multi-turn GRPO 就失去了依托。

---

## 为什么它对面试重要

Search Agent 项目和普通 QA 项目最大的区别之一，就是它真的依赖一个“可被调用的外部检索环境”。

M3 让你能清楚说明：

- 模型检索的对象是什么
- 检索服务如何被训练脚本调用
- 为什么要先做 BM25 baseline，再扩展到 Dense / RRF

---

## 范围

### M3 要完成的事

- 本地小语料检索服务 smoke test
- BM25 baseline 构建
- 检索服务接口稳定化
- 为后续混合检索预留扩展位

### M3 不做的事

- 不一开始就追求最强混合检索
- 不要求本地完成全量 Wikipedia 索引
- 不把检索服务做成复杂平台工程

---

## 推荐推进顺序

### 第一层：本地可验证 baseline

- 用小语料构造 `corpus.jsonl`
- 启动本地检索服务
- 能返回 top-k 结果
- 训练脚本能成功请求该服务

### 第二层：可信 BM25 baseline

- 在远端构建正式 BM25 索引
- 形成一个真正可用于训练的检索基础环境

### 第三层：增强版检索

- 加入 Dense Retrieval
- 加入 RRF 融合
- 对比 BM25-only 与 Hybrid 检索效果

---

## 关键输出物

- 本地可运行的检索服务
- BM25 baseline
- 为 Dense / RRF 扩展预留的检索接口
- 可被 rollout 环境直接调用的 API

---

## 通过标准

满足以下条件即可认为 M3 完成：

- 本地 retriever smoke test 跑通
- 检索服务接口固定下来
- 训练脚本能成功调检索服务
- 至少有一个可信 baseline 检索器可用于后续训练

---

## 面试讲法

M3 对外可以这样讲：

> 我没有把“检索”停留在概念层，而是实现了一个可被训练脚本直接调用的本地检索服务。项目先以 BM25 建立可信 baseline，再扩展到 Dense Retrieval 和 RRF，从而保证方法改进有明确对照。
