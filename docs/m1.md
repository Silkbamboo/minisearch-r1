# M1：环境与工程工作流

## 本模块回答什么问题

M1 回答的是：

> 这个项目有没有一套稳定、专业、可复用的开发与训练工作流？

它不是算法模块，但它决定了后面所有实验能不能有序推进。

---

## 为什么它对面试重要

对于大模型算法实习岗位，面试官通常不只看“你会不会写训练脚本”，还会看：

- 你是否理解本地开发与远端训练分工
- 你是否有 Git / GitHub 协作意识
- 你是否能在资源受限条件下做工程取舍

一个清晰的 M1，会直接提升项目的可信度。

---

## 范围

### M1 要完成的事

- 建立本地开发环境
- 建立 Git / GitHub 同步流程
- 建立 AutoDL 执行流程
- 明确本地与远端的职责边界
- 搭好项目骨架与依赖拆分

### M1 不做的事

- 不做正式训练
- 不做全量语料处理
- 不做实验结果分析

---

## 关键输出物

- 项目仓库初始化完成
- `requirements-local.txt` 与 `requirements-server.txt`
- 本地可运行的基础脚本入口
- GitHub 作为代码真相源
- AutoDL 作为高资源执行环境

---

## 通过标准

满足以下条件即可认为 M1 完成：

- 本地可以正常开发、提交、推送代码
- 服务器可以通过 Git 拉取最新代码
- 项目目录结构稳定
- 本地 smoke test 与远端 full train 的职责边界明确

---

## 面试讲法

M1 对外可以这样讲：

> 我采用本地开发、GitHub 同步、远端训练的标准工作流。本地只做代码开发和低资源验证，正式训练与大规模索引构建都放在远端完成。这样既节省成本，也保证实验链路清晰可复现。
