# M2：数据准备与本地小样本验证

## 本模块回答什么问题

M2 回答的是：

> 在不依赖大显存和大磁盘的前提下，这个项目的数据链路能不能先在本地跑通？

M2 的重点不是“处理完整 Wikipedia”，而是先把 **数据格式、样本组织、训练输入** 验证清楚。

---

## 为什么它对面试重要

Search Agent 项目如果只有训练脚本，没有清楚的数据链路，面试官会很难判断你是否真正理解任务。

M2 的价值在于让你能够清楚说明：

- benchmark 数据怎么来
- SFT 数据怎么构造
- GRPO 数据怎么组织
- 课程学习数据怎么拆分
- 检索语料如何先做本地开发版，再做远端全量版

---

## 范围

### M2 要完成的事

- 下载 benchmark 小样本
- 生成本地开发用的训练数据
- 生成本地开发用的 corpus
- 生成按 hop 拆分的课程学习数据
- 验证 `sft_train.jsonl` / `grpo_train.jsonl` / `corpus.jsonl` 可用

### M2 不做的事

- 不在本地处理全量 KILT Wikipedia
- 不在本地构建全量 Dense 索引
- 不在本地进行正式训练

---

## 关键输出物

本地开发阶段，M2 的核心输出是：

- `data/processed/sft_train.jsonl`
- `data/processed/sft_eval.jsonl`
- `data/processed/grpo_train.jsonl`
- `data/processed/grpo_eval.jsonl`
- `data/processed/hops/*.jsonl`
- `data/processed/corpus.jsonl`

这些文件的意义不是“最终实验数据”，而是：

> 用最小成本证明数据链路设计成立。

---

## 建议流程

### 本地阶段

优先使用 benchmark 子集或 fixture：

1. 下载小样本 benchmark
2. 生成 SFT / GRPO 数据
3. 构建本地 dev corpus
4. 验证训练脚本 dry-run 或 smoke-test 能读取这些数据

### 远端阶段

当本地链路验证通过后，再迁移到远端做：

- 全量语料准备
- 全量索引构建
- 正式训练数据版本整理

---

## 通过标准

满足以下条件即可认为 M2 完成：

- 数据处理脚本在本地可执行
- 训练脚本能读到生成的数据
- 本地小语料足以支撑检索和训练 smoke test
- 本地开发不依赖大硬盘和大显存

---

## 面试讲法

M2 对外可以这样讲：

> 我把数据准备拆成了本地小样本验证和远端全量执行两层。本地先验证格式转换、课程学习拆分和训练输入链路，确认逻辑正确后，再把全量语料处理放到远端执行。这保证了项目既可复现，也能控制开发成本。
