# M5：Multi-turn 环境与 Reward 设计

## 本模块回答什么问题

M5 回答的是：

> 模型在训练时，是否真的能“边生成、边搜索、边继续推理”，并且这个过程能被合理奖励？

这部分是项目的核心分水岭，也是它区别于普通 SFT 或单轮 QA 项目的关键。

---

## 为什么它对面试重要

如果你能把 M5 讲清楚，面试官通常会认为你对 Search Agent 训练已经进入“真正理解”的层面，因为这里涉及：

- 多轮 rollout 控制
- 外部检索结果注入
- retrieved token masking
- reward 设计与 reward hacking 风险

---

## 范围

### M5 要完成的事

- 设计 multi-turn 搜索交互协议
- 实现 rollout 环境
- 构造 retrieval mask
- 设计并实现 reward 函数
- 为后续 GRPO 提供可调用的环境与打分逻辑

### M5 不做的事

- 不在本模块内完成正式大规模 GRPO 训练
- 不把 reward 设计扩展到复杂 Reward Model

---

## M5 的关键点

### 1. 搜索不是一次性调用

模型需要经历：

1. 生成推理内容
2. 输出搜索请求
3. 接收检索结果
4. 继续推理
5. 最终回答

因此这里的训练环境不再是“普通 generate 一次完事”。

### 2. 检索内容不能参与梯度学习

注入的 `<information>` 内容属于环境返回，不属于模型应该学习生成的部分，因此需要做 retrieved token masking。

### 3. Reward 既要有结果信号，也要有过程信号

这正是过程奖励存在的意义：

- 答案是否正确
- 格式是否正确
- 搜索是否有效

---

## 关键输出物

- `search_env.py`
- `rewards.py`
- retrieval masking 逻辑
- 本地可验证的 rollout / reward smoke test

---

## 通过标准

满足以下条件即可认为 M5 完成：

- 一次 multi-turn rollout 可以完整跑完
- 检索结果能被正确注入上下文
- reward 函数能输出合理值
- injected token 不参与训练 loss

---

## 面试讲法

M5 对外可以这样讲：

> 项目最难的部分不是“把模型训起来”，而是把 multi-turn 搜索环境和训练目标正确耦合起来。我的重点工作包括检索结果注入、retrieved token masking，以及将答案质量和搜索质量组合成可训练的 reward 信号。
