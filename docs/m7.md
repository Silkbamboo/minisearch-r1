# M7：评测、消融与面试展示材料

## 本模块回答什么问题

M7 回答的是：

> 这个项目最后如何转化成能打动面试官的证据，而不是只留下一堆训练脚本和 checkpoint？

M7 的重点不是“再多写代码”，而是把前面 M1-M6 的成果整理成**可展示、可比较、可讲述**的结果。

---

## 为什么它对面试重要

对面试官来说，一个项目的价值通常最终体现在三类证据上：

- 数字结果
- 关键对比
- 对失败与局限的理解

M7 就是把这些证据系统化。

---

## 范围

### M7 要完成的事

- 统一评测主实验结果
- 选择最关键的 2-3 组消融
- 形成 case study 与 failure analysis
- 打磨 README 与 docs
- 准备面试讲述材料

### M7 不做的事

- 不为“完整性”强行补跑所有消融
- 不追求论文级排版和巨量图表

---

## 推荐结果组织方式

### 1. 主实验

至少形成以下对比：

- Base model
- Vanilla RAG
- SFT-only
- MiniSearch-R1

### 2. 关键消融

每组消融对应一个明确假设，防止消融变成"看看效果如何"：

| 消融 | 假设 | 核心对比指标 |
|------|------|------------|
| w/o 课程学习 | H1：课程学习使 1.5B 模型在有限 budget 内更快建立非零 reward 信号 | 前 200 步 reward 曲线 + 最终 EM |
| BM25-only | H2：Dense+BM25 提升多跳题第二步搜索 recall，进而提升最终 EM | Hit@3 + EM/F1 |
| F1-only reward | H3：过程奖励防止冷启动阶段 GRPO advantage 归零 | 前 200 步 reward 均值 |

### 3. 案例分析

建议至少准备三类案例：

- 搜索明显帮助了回答的案例
- 搜索没帮上忙的失败案例
- 课程学习或过程奖励带来改善的代表案例

---

## 关键输出物

- 主实验结果表
- 关键消融结果表
- 至少 2-3 个 case study
- 面试版 README
- 项目讲述提纲

---

## 通过标准

满足以下条件即可认为 M7 完成：

- 项目有一组主结果可以展示
- 至少有若干关键对比可以说明方法价值
- README 能独立讲清项目
- 你能在面试中顺畅讲完整个项目

---

## 面试讲法

M7 对外可以这样讲：

> 我最终不是只给出一个模型结果，而是整理出一条完整证据链：先有 baseline，再有方法改进，再用主实验和关键消融说明这些改进为什么有效，同时补充失败案例分析，让整个项目既有结果，也有解释。
